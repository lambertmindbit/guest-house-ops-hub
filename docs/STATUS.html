Guest House Operations Hub — Project Status

Self-hosted, mobile-first ops hub for a single small property · Next.js 15 · Prisma · PostgreSQL (Supabase) · Vercel · As of 2026-06-03

Bottom line: all six planned milestones are built, tested, and live in production on main. Several once-deferred items now have their app-side groundwork built behind clean seams (off by default) so they aren’t a last-minute scramble — see the Deferred section. CI (lint + build + 25 tests on an ephemeral Postgres) gates every merge.

✓ Delivered phases

1

Operations core + Admin Delivered

Bookings (create / edit / cancel) with database-enforced no-double-booking; unified colour-coded calendar (rooms × dates, week / 2-week / month views); Today dashboard; guest list & search; housekeeping (derived “needs cleaning” + manual flag); single-owner login; installable PWA. Plus a full Settings/Admin area: add / edit / archive rooms, manage room types, channels & commissions, the property profile, and maintenance blocks.

2

Pricing engine + rate calendar Delivered

Rule-based nightly rate suggestions — weekend, season/holiday, lead-time (early-bird / last-minute), and occupancy — compounded then clamped to each room type’s floor/ceiling. Rate calendar with tap-to-pin overrides; suggested price pre-fills new bookings. Advisory only: never pushed to OTAs, never rewrites saved bookings.

3

Check-in / check-out Delivered

Live arrival/departure tracking on each booking (check-in / check-out / undo), surfaced on the reservation detail and the Today board.

4

Guest CRM Delivered

Guest profiles with stay history, repeat-guest badge, lifetime value, ID (text), notes, and a blacklist toggle that warns at booking time.

5

Finance → real net profit Delivered

Per-channel revenue and commission, plus expense tracking so Finance shows true net profit (gross − commission − expenses).

6

Invoices + CSV export Delivered

Printable guest invoices (browser Print → PDF, using the property profile) and dependency-free Bookings / Payments CSV exports for the accountant.

▲ Known concerns & tech debt

Tracked with file/line references in .planning/codebase/CONCERNS.md. None are blockers; the most important to understand are flagged ◆.

AreaWhat & why it mattersSeverity
Migration discipline The correctness core uses generated DATERANGE columns + a GiST exclusion constraint that Prisma’s diff engine mangles. Migrations must be created with npm run db:migrate:new, never prisma migrate dev. Most load-bearing, easiest-to-trip area. ◆ High
Overlap-error detection The friendly “dates unavailable” 409 relies on sniffing the Postgres error string (23P01 / constraint name) in src/lib/db-errors.ts. A rename or Prisma change could silently break it → keep tests/conflict.test.ts green. ◆ Medium-High
Test breadth Correctness core (conflicts, availability) and pure pricing math are tested; API route handlers and the quoteRoomType data wrapper are not. Route-level regressions wouldn’t be caught by CI. ▲ Medium
Money math Amounts are whole-rupee JS number with Math.round; per-booking commission is rounded before summing. Fine today; strict accounting would need integer-paise / decimal arithmetic end to end. ▲ Low
Auth Single-owner login: constant-time compare + signed httpOnly cookie. Login is now rate-limited (10/IP/5 min). The env password is plaintext (acceptable for single-owner self-host); no per-user roles yet. ▲ Low (by design)
Pricing rate-calendar queries The rate calendar re-fetches global config (policy/seasons/property) once per room type — a mild N+1. Negligible at this scale; batch it only if room types grow large. ▲ Low
Resolved during handoff Already fixed: CSV formula-injection guard; db:migrate footgun (now apply-only prisma migrate deploy); untracked tsconfig.tsbuildinfo; login rate-limiting added. Fixed

→ Suggested first tasks (for the new team)

  1. Read in order: docs/ARCHITECTURE.md (the correctness core) → docs/CONTRIBUTING.md (migration safety) → docs/SETUP.md (get a local env up and run npm test).
  2. Close the biggest test gap: add a route-level test for the create-overlap → 409 path, plus a housekeeping-derivation test. These cover the highest-risk untested code.
  3. Harden the constraint guard: add a standalone “exclusion constraint exists” check to the suite so an accidental migration can’t silently drop it.
  4. Skim the debt list: .planning/codebase/CONCERNS.md before any large refactor.
  5. Optional privacy hardening: move the single shared iCal token to per-feed tokens so rotating one feed doesn’t invalidate all OTA subscriptions.
The one rule to internalise first: never run prisma migrate dev. Create migrations with npm run db:migrate:new <name>; apply them with npm run db:migrate. This protects the generated columns and the no-double-booking constraint that the whole app’s correctness depends on.
