⚙ Enabling the groundwork features (setup)

Three features are built but dormant until configured. Here is exactly how to switch each on. Set production values in Vercel → Project → Settings → Environment Variables (and in your local .env for testing). None require code changes except optional parser tuning.

A · Automated OTA email ingestion 🟡 needs wiring

Works today with zero setup by pasting an email into the Inbox screen. The steps below make import automatic (emails land in the Inbox by themselves). Either way, parsed bookings are always staged for your review — nothing is ever auto-booked. Full technical walkthrough follows.

How the webhook works (the contract you build against)

There is one endpoint. Your job is simply to deliver each OTA email's text to it. It lives outside the owner login (so a forwarder can reach it) and is gated by its own shared secret, INGEST_TOKEN.

FieldValue
Method & URLPOST https://YOUR-APP.vercel.app/api/ingest/email
AuthHeader x-ingest-token: YOUR_INGEST_TOKEN  (or ?token=YOUR_INGEST_TOKEN on the URL)
Bodyapplication/json — { "raw": "FULL EMAIL TEXT" }  (headers + body, as plain text — the parser greps it)
Success201 { "data": { "id": "...", "status": "pending" } } → now visible in the Inbox
Rejected401 { "error": "unauthorized" } (token missing/wrong, or INGEST_TOKEN unset) · 422 (body isn't { raw })
Retry-safeDe-duplicated by the OTA reference — the same confirmation arriving twice won't create two review items.

Step 1 — Create the secret token

  1. Generate one: openssl rand -hex 32 (any long random string works).
  2. In Vercel → Project → Settings → Environment Variables, add INGEST_TOKEN = that value, for Production (and Preview if you test there). Redeploy so it takes effect.
  3. Add the same line to your local .env if you want to test locally. Until INGEST_TOKEN is set, the endpoint safely rejects everything with 401.

Step 2 — Deploy the forwarder (recommended: Cloudflare Email Worker)

Already built and ready in the repo at integrations/cloudflare-email-worker/ (Worker + wrangler.toml + README) — you don't write this, you just fill in the token + URL and wrangler deploy. Cloudflare Email Routing is free and runs the script on each received email, so it can POST exactly the JSON the endpoint wants. (Any other method — Mailgun/SendGrid inbound parse, an IMAP poller — is fine too; it just has to end in the same POST { raw } call.) No domain / no Cloudflare? A ready-to-use Gmail Apps Script version is in integrations/gmail-apps-script/ — it runs inside your Google account on a timer, same token + webhook, zero DNS. The Cloudflare Worker source for reference:

  1. You need a domain on Cloudflare. In Cloudflare → your domain → Email → Email Routing, enable it and verify the records.
  2. Create an address, e.g. bookings@yourdomain.com, and set its action to Send to a Worker.
  3. Create the Worker with this code, and add INGEST_TOKEN as a Worker Variable/Secret (same value as in Vercel):
export default {
  async email(message, env) {
    // The full raw email (headers + body) as text
    const raw = await new Response(message.raw).text();

    await fetch("https://YOUR-APP.vercel.app/api/ingest/email", {
      method: "POST",
      headers: {
        "content-type": "application/json",
        "x-ingest-token": env.INGEST_TOKEN,
      },
      body: JSON.stringify({ raw }),
    });
  },
};
  1. Get the OTA emails to that address. In the mailbox where Booking.com / Agoda / MakeMyTrip confirmations arrive, add a filter/forwarding rule that forwards those confirmations to bookings@yourdomain.com. (Or point the OTA's notification email straight there.)

Step 3 — Smoke-test the endpoint

Before wiring real mail, confirm the endpoint + token work with a fake email:

curl -X POST https://YOUR-APP.vercel.app/api/ingest/email \
  -H "content-type: application/json" \
  -H "x-ingest-token: YOUR_INGEST_TOKEN" \
  -d '{"raw":"Booking.com\nGuest name: Test Guest\nCheck-in: 2026-07-12\nCheck-out: 2026-07-14\nRoom: Deluxe\nTotal: ₹4200\nConfirmation: TEST-123"}'

Expect 201 with an id, and the item to appear in the Inbox screen as pending. A 401 means the token doesn't match; a 422 means the body wasn't { "raw": "..." }.

Step 4 — Tune the parser against real emails

The reader in src/lib/email-parse.ts is best-effort scaffolding — it fills what it can and leaves the rest blank for you to correct on the review screen. Tune it once against real samples so fewer fields need correcting. It extracts:

FieldHow it's found
sourceKeyword match — "booking.com" / "agoda" / "makemytrip" → else Unknown
otaRefA "booking/reservation/confirmation/itinerary number" label followed by a code
guestNameA Guest name: / Name: / Booker: label
guestPhoneA Phone:/Mobile: label, else a long phone-like number
checkIn / checkOutThe Check-in: / Check-out: lines (dates like 2026-07-12, 12 Jul 2026, July 12, 2026)
roomTypeHintA Room type: / Room: / Accommodation: label
amountA Total/Amount/Price figure (handles ₹ / INR / Rs and commas)
  1. Paste 2–3 real Booking.com / Agoda / MakeMyTrip confirmations into the Inbox and note which fields come out wrong/blank.
  2. Adjust the matching regex for just those fields in src/lib/email-parse.ts (each field is isolated, so changes are low-risk).
  3. Add a real (anonymised) sample as a fixture in tests/email-parse.test.ts and run npm test so the format stays covered.

Step 5 — Verify end-to-end

Forward a real test confirmation to bookings@yourdomain.com → it appears in the Inbox as pending → review/correct the fields, pick the room → Create booking (which goes through the normal POST /api/reservations, so it's conflict-checked like any booking). Re-sends are ignored thanks to the OTA-reference de-dupe.

Safety: automation only fills the Inbox — it never creates a booking on its own. You always do the final review + Create booking step, so a mis-parsed email can't corrupt the calendar.

B · Guest ID document upload 🟡 needs a bucket

Until configured, the guest profile shows a “not configured” hint; everything else works.

  1. In Supabase → Storage → New bucket, create a bucket named guest-ids and keep it Private (do not make it public — IDs are sensitive).
  2. In Supabase → Project Settings → API, copy the Project URL and the service_role key.
  3. Set in Vercel: SUPABASE_URL = the project URL, SUPABASE_SERVICE_ROLE_KEY = the service_role key, and optionally SUPABASE_ID_BUCKET = guest-ids. Redeploy.
  4. Verify: open a guest → Upload a JPG/PNG/WEBP/PDF (≤ 5 MB) → View opens it via a short-lived signed URL; Remove deletes it.
Security: the service-role key bypasses row-level security, so it is server-only — never put it in a NEXT_PUBLIC_* var. Only the API routes use it; the browser only ever receives short-lived signed URLs.

C · Login rate-limiting ● already on

No setup needed — active out of the box (10 attempts / IP / 5 min). It is in-memory, so on serverless it is best-effort per instance. Optional upgrade: for a strict global cap, back src/lib/rate-limit.ts with a shared store (e.g. Upstash Redis).

▲ Known concerns & tech debt

Tracked with file/line references in .planning/codebase/CONCERNS.md. None are blockers; the most important to understand are flagged ◆.

AreaWhat & why it mattersSeverity
Migration discipline The correctness core uses generated DATERANGE columns + a GiST exclusion constraint that Prisma’s diff engine mangles. Migrations must be created with npm run db:migrate:new, never prisma migrate dev. Most load-bearing, easiest-to-trip area. ◆ High
Overlap-error detection The friendly “dates unavailable” 409 relies on sniffing the Postgres error string (23P01 / constraint name) in src/lib/db-errors.ts. A rename or Prisma change could silently break it → keep tests/conflict.test.ts green. ◆ Medium-High
Test breadth Correctness core (conflicts, availability) and pure pricing math are tested; API route handlers and the quoteRoomType data wrapper are not. Route-level regressions wouldn’t be caught by CI. ▲ Medium
Money math Amounts are whole-rupee JS number with Math.round; per-booking commission is rounded before summing. Fine today; strict accounting would need integer-paise / decimal arithmetic end to end. ▲ Low
Auth Single-owner login: constant-time compare + signed httpOnly cookie. Login is now rate-limited (10/IP/5 min). The env password is plaintext (acceptable for single-owner self-host); no per-user roles yet. ▲ Low (by design)
Pricing rate-calendar queries The rate calendar re-fetches global config (policy/seasons/property) once per room type — a mild N+1. Negligible at this scale; batch it only if room types grow large. ▲ Low
