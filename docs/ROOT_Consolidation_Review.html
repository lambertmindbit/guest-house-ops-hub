1What you already have that's worth protecting

The Ops Hub is a genuinely strong foundation — not a prototype to throw away. Before adding anything, here's what must stay sacred, because the consolidation only works if these survive:

  • The correctness core. DB-enforced no-double-booking (GiST exclusion constraint, half-open daterange, derived availability). Your own handoff already notes it is keyed per room_id, so it stays correct under multi-tenancy untouched. Every booking — human, agent, or OTA — must flow through this.
  • The staging-seam pattern. InboundBooking + the token-gated /api/ingest/email webhook is exactly the template for plugging in an external producer (an AI agent) without giving it write access. The escalation module copies this pattern.
  • The domain-layer discipline. Logic in src/lib/*, reused by both Server Components and routes; {data}/{error} + Zod; migration safety helper. This is what makes the agents safe to bolt on — they call the same guarded paths a human does.
  • Local ownership. Self-hosted, owner-controlled data. The Lawei response rates "my data stays mine" as very important and privacy concern 5/5 — this architecture is the moat the DPR sells.

4The integration architecture

The mental model that makes one product out of all the pieces. The PMS is the system of record. The agents are a layer on top that transacts through it.

Guests / drivers reach you via Web chat · WhatsApp · phone · OTA email · iCal
↓

Deterministic core — Postgres + the GiST no-double-booking constraintThe single source of truth. Impossible to double-book, no matter who asked.
↕   anything an agent can't safely do →

Five principles that keep it coherent

  1. The PMS is the only system of record. Agents and OTAs are inputs; the database is truth. No agent keeps its own booking state.
  2. Agents read freely, write through the front door. An agent calls /api/availability or /api/pricing/quote to answer a guest, and creates a booking via POST /api/reservations — inheriting the 409 conflict check. It never writes the DB directly.
  3. Sensitive actions are escalations, not agent actions. Cancel / refund / blacklist → filed to the HITL queue; a human commits them through the guarded route. Matches Lawei's "cancellation must need approval."
  4. The escalation queue is the universal seam. All three agents speak one contract (POST /api/agent/escalations). Build it once; every agent plugs in the same way.
  5. The community network is a cross-tenant tier. Overflow, scam lists, bad-guest alerts, shared availability are between properties — they only exist once properties are first-class (the gate).

5The shared gate: multi-tenancy + identity

Four things you want all depend on the same prerequisite:

  • "Any homestay owner" → many properties → tenancy.
  • Staff roles (Lawei: money owner-only) → real accounts + roles.
  • Community network → properties must have stable identities to refer between, share availability, and attribute scam/bad-guest reports.
  • Direction B (channel-manager SaaS) → multi-tenant by definition.

So this isn't "the ROOT project" vs "Direction B" — they converge on one Phase 1. Do tenancy + auth + identity once, carefully, and all four unlock. Your PRODUCT-ROADMAP-CM.html already scoped this well; the only change is recognising it's the gate for the agents and the community network too, not just the SaaS pivot.

The good news (your own handoff says it): the correctness core survives tenancy unchanged because the constraint is keyed per room_id and a room belongs to one property. Tenancy is invasive (a property_id on every scoped table + a filter on every query) but it does not threaten the one thing that's hard to get right.

7Honest risks & what I would not do yet

  1. Don't build the community network on a single-tenant base. It's tempting because demand is high — but every referral/scam/availability feature assumes property identities. Retrofitting tenant isolation later risks data leaks between businesses. Gate first.
  2. Don't let any agent write to the DB directly. The moment an agent bypasses /api/reservations, the no-double-booking guarantee is gone. Reads anywhere; writes only through the guarded paths; sensitive actions via escalation.
  3. WhatsApp is the long pole, not the code. BSP approval + template-message rules + per-message cost have real lead time and recurring cost. The #1 pilot pain depends on it — start the provider process in Phase 0, not when you're ready to code it.
  4. Scam list & bad-guest alerts carry legal risk. Defamation / privacy exposure if a property is wrongly flagged. The questionnaire deliberately asks who can report and what evidence is required — design that governance before building the feature, not after.
  5. n = 1. One questionnaire isn't product-market fit. Get the other three pilots' responses before locking community-network scope.
  6. Money is whole-rupee number math today. Fine for the homestay; revisit (integer paise / Decimal) before the Console reports financials others rely on, and before any billing in Direction B.
