Product Roadmap — From Ops Hub to Channel-Manager SaaS

Direction B: turn the single-property Guest House Ops Hub into a multi-tenant channel-manager product we sell to other hotels. Grounded in the current codebase. Companion to docs/CHANNEL-MANAGERS.html and docs/CHANNEL-SYNC.html.
The shift in one line: we already have a solid per-property PMS core (calendar, derived availability, no-double-booking, pricing, guests, payments). To become a channel-manager product we add two new layers on top: (1) multi-tenancy (serve many hotels) and (2) a distribution layer (push availability to OTAs and pull their bookings in, via Channex to start). Everything else is reuse.

2The phased roadmap

0

Foundations & decisions

~1–2 weeks · before any code

  • Tenancy model: recommended — single database with a tenant_id on every row (shared schema, row-level scoping). Simpler than schema-per-tenant at this scale.
  • Auth stack: pick one (Auth.js/NextAuth, Clerk, or Supabase Auth) for multi-user + roles.
  • Channex sandbox: get developer/sandbox access to validate the distribution path early.
  • Business setup: legal entity + the OTA partner relationships (even via Channex, you sign up properties under your account).
1

Multi-tenancy refactor the gate

The hard prerequisite — everything else depends on it

  • Add an Organization (account) and Property (a hotel) model; a property owns its rooms, reservations, blocks, pricing, guests.
  • Add property_id / org_id to the scoped tables and filter every query by the current tenant (availability, calendar, dashboard, finance…).
  • Keep the GiST exclusion constraint as-is (per room_id → already tenant-safe).
  • Make PropertySettings per-property instead of a singleton.
2

Real auth & user management

  • Replace the single env login with accounts; users belong to an organization with roles (owner / manager / staff).
  • Invitations, password reset, session management.
  • Every request resolves which tenant the user is acting on (drives the Phase 1 scoping).
3

Distribution core — one OTA, end to end the product value

  • Integrate Channex: map a property's room types & rate plans; push availability/rates/inventory out.
  • Receive a Booking.com booking via Channex webhook → create a reservation (reusing the no-double-booking core) → confirm back.
  • Handle cancellations & modifications idempotently (extend InboundBooking).
  • Prove the loop with one pilot hotel before building onboarding/billing.
4

Self-serve onboarding

  • Signup → create organization & property → add rooms/room types.
  • Connect OTAs (through Channex) → map room types/rate plans → go live.
  • Guardrails so a half-configured property can't push bad data.
5

Billing & plans

  • Subscription billing per property (Razorpay / Cashfree for India, Stripe for global) — hosted pages, no card storage (see PCI note in CHANNEL-MANAGERS).
  • Plan tiers (rooms / OTAs / features), trials, dunning.
6

Vendor back-office & reliability

  • Admin to manage tenants; a connection-health dashboard (which property↔OTA links are live/erroring).
  • Reconciliation reports, audit logs, monitoring/alerting, per-tenant rate limits, a security review.
  • Then scale: more OTAs, and later direct certification with the biggest ones to cut per-property cost.

3Smallest first proof (do this before the rest)

Before building onboarding, billing, or polish, prove the thin vertical slice:
  1. Hand-onboard one pilot hotel (manual setup, no signup flow yet).
  2. Push its availability to Booking.com via Channex.
  3. Make a real Booking.com booking → it flows back into the app → no double-booking, cancellation works.
If that loop is solid and reliable, the business is real and the rest is (a lot of) product work. If it's flaky, fix that before anything else — it's the whole value proposition.

4Honest risks & sequencing notes

  • Multi-tenancy is invasive — it touches the schema and nearly every query. Do it first and carefully; retrofitting tenant isolation later is painful and risky (data leaks between hotels).
  • This is a product + a business, not just code: SLAs, support for paying hotels, per-OTA mapping quirks, reconciliation correctness. Budget for operations, not just build.
  • Channex is the de-risk — it lets us validate demand and ship in weeks without certifying with every OTA. Certify directly only once volume justifies it.
  • Keep the correctness core sacred — the DB-level no-double-booking guarantee is the single most valuable thing we already have; every OTA booking must flow through it.
