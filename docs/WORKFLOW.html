01The system, top to bottom

Five layers. The database is the only place truth lives — everything above it is a way of reading or carefully proposing a change to it.

Layer 1 — Client
Guest's phone · Owner's console
The public chat widget and the in-app assistant are the same protocol, different mode
HTTPS, NDJSON stream
Layer 2 — Next.js (Vercel)
The OTA app — ships NO LLM SDK
Pages render server-side from Postgres; a small proxy streams chat to the agent below
proxy · falls back to a stub if unreachable
Layer 3 — assistant-agent (Cloud Run)
Python · Google ADK · Gemini
guest_agentowner_agent pinned to 1 instance
/api/agent/* · token-gated, fail-closed
Layer 4 — the seam, back in the app
Same domain functions every screen uses
The agent reads/writes through the identical code path as the owner's own forms — no shortcut
Layer 5 — PostgreSQL (Supabase)
The one source of truth
A GiST exclusion constraint makes double-booking physically impossible, for anyone, including the AI
Why two services, not one

The agent is a separate Python process from the Next.js app, on purpose — the app never imports an LLM SDK. If the agent is down, unreachable, or simply not configured, the app quietly serves a canned stub instead. A guest can always see rooms and prices; a broken AI never breaks the booking system underneath it.

02A booking, step by step

Following one real conversation — a guest asking for a room — from the first message to the row landing in Postgres. The AI's job ends well before the database write.

Guest
Types "a room for 4 people, 13 to 15 July" into the chat widget.
guest_agent (thinks, then acts)
A thinking planner runs first, then the model picks a tool: dates are present, so this is booking territory — it calls check_availability(check_in, check_out, guests=4), never the browse tool.
Tool → seam → database
The tool calls GET /api/agent/rooms/availability — the exact same derived-availability query the calendar page uses. Rooms are filtered to maxOccupancy ≥ 4 and sorted smallest-fit-first.
Guest sees
A card for each fitting room — photo, price, facing/view, amenities. The guest taps Book on 301.
Deterministic flow begins — no LLM from here
Tapping Book opens a fixed name/phone form. This is ordinary application code, not a model decision — the same two fields, every time.
Guest confirms
/confirm re-checks availability one more time, then files a booking-request escalation — a guest's booking is never written directly, it's proposed to the owner.
Owner taps Approve
The escalation queue's Approve & book action runs the guarded write — the identical tx.reservation.create transaction the manual booking form uses.
Postgres
The GiST exclusion constraint checks the room + date range against every other confirmed stay. Clear → row written. Someone got there first → 409, "just taken" — surfaced as a friendly message, never retried automatically.
The invariant this protects

Two confirmed reservations for the same physical room cannot overlap — not because the AI is careful, but because the database refuses the write. Every path in this diagram — guest chat, owner console, the manual form — ends at the same guarded transaction.

03Two agents, one boundary

The public widget and the owner console are served by separate agents with separate memory — not one agent that checks a permission flag. A guest's conversation and an owner's conversation never share state.

guest_agent

public widget · anonymous
  • list_rooms — browse with photos, no dates needed
  • check_availability — date + party-size filtered
  • quote_room
  • propose_booking — shows a confirm card, never writes
  • answer_faq — owner-managed Q&A only
  • request_booking_change — cancel/modify → always escalated
  • pass_to_property — anything else, filed for a human

owner_agent

in-app console · authenticated owner
  • everything guest_agent has, plus:
  • daily_briefing — occupancy, arrivals, departures now
  • open_requests — the escalation queue
  • block_room — hold a room out of service
  • resolve_request — act on a queued item
  • business_summary — revenue / finance snapshot
enforced at startup — the process refuses to boot if this line is crossed
Provable, not just intended

A structural check (guardrails.assert_tool_isolation()) runs when the agent process starts: if an owner-only tool is ever accidentally wired onto guest_agent, the service fails to start rather than quietly exposing owner data to the public widget. This is checked by a test that deliberately tries the mistake and confirms it's caught.

If every attempt fails, the guest gets one honest sentence — "I hit a hiccup — please ask again" — never a stack trace, never dead air. If an attempt streams a partial answer before erroring, that partial is kept rather than restarted on a different model (which could contradict what was already said).

05Answering a question

Facility questions ("is there a pool?", "any photos?") never touch the booking tools — and the agent is structurally prevented from answering with the wrong facility's information.

Guest
"is there a pool?"
answer_faq(topics=["pool"])
The tool fetches the owner's FAQ list and splits it into matched_faqs (word + synonym matched against this topic) and other_faqs — the model only ever reasons over the split, never the raw list.
If nothing matches
The tool returns an explicit instruction: "nothing answers this — say you don't know and file it with pass_to_property. Do NOT substitute a different facility's answer."
If a photo FAQ matches
A gallery card renders alongside the text — photo URLs never reach the model, only the card. The model can describe what's in an answer but can't hallucinate a caption for an image it never saw.

06Where things run

Two platforms, two very different scaling stories — because they hold state differently.

Vercel
Next.js app · stateless
Scales to
Many concurrent instances — safe, no shared memory needed
Deploys on
Every push to main, automatically
Migrations
prisma migrate deploy runs in the build step
Database
Supabase Postgres, pooled connection
Cloud Run
assistant-agent · stateful
Scales to
exactly 1 instance — pinned, re-checked after every deploy
Why pinned
Conversation + any in-flight booking live in this process's memory (ADK's InMemorySessionService) — a 2nd instance would serve a follow-up message from different memory and lose the pending booking mid-flow
Deploys on
Manual — gcloud run deploy, then re-pin max-instances
Region
asia-south1
The one deploy step that's easy to forget

A fresh Cloud Run deploy can silently reset the instance cap. Every deploy is followed by gcloud run services update ota-guest-agent --max-instances=1 and a check that it stuck — skipping this is the single most likely way to reintroduce the lost-booking bug.
